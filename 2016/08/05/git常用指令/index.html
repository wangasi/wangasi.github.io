git常用指令

发表于 2016-08-05 | 分类于 工具 |

创建版本库

  1. git init 把当前目录变成git可以管理的仓库。
  2. git add readme.txt 添加文件到仓库。
  3. git commit -m “描述内容” 提交文件到仓库。
  4. git log –graph 查看分支合并图。

版本回退

  1. 1) git log 查看从最近到最远的提交日志
    2) git log –pretty=oneline
    3) git reflog 用来记录你的每一次命令。

  2. 1) git使用HEAD表示当前版本，上一个版本就是HEAD^，上上一个版本就是HEAD^^，
    当然往上100个版本写100个^比较容易数不过来，所以写成HEAD~100。
    2) git reset –hard HEAD^。
    3) cat readme.txt 查看文件内容。
    4) git reset –hard 3628164 “3628164”想要回到的未来的某个版本号，前几位就可以，git会自动去找。

小结
a. HEAD指向的版本就是当前版本，因此，Git允许我们在版本的历史之间穿梭，
   使用命令git reset --hard commit_id。
b. 穿梭前，用git log可以查看提交历史，以便确定要回退到哪个版本。
c. 要重返未来，用git reflog查看命令历史，以便确定要回到未来的哪个版本。

工作区和暂存区的区别

  1. git add实际上是把文件修改添加到暂存区。
  2. git commit提交更改，实际上是把暂存区的所有内容提交到当前分支。
  3. git status 查看状态。
  4. git diff HEAD – readme.txt 查看工作区和版本库里面最新版本的区别。
  5. git checkout – file 可以丢弃工作区的修改。没有“–”就成立切换分支。
    这里有两种情况：
1）一种是readme.txt自修改后还没有被放到暂存区，现在，撤销修改就回到和版本库一模一样的状态；
2）一种是readme.txt已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态。
总之，就是让这个文件回到最近一次git commit或git add时的状态。
  6. git reset HEAD file 可以把暂存区的修改撤销掉，重新放回工作区。git reset命令既可以回退版本，也可以把暂存区的修改回退到工作区。
  7. git rm file 删除文件。
小结
场景1：当你改乱了工作区某个文件的内容，想直接丢弃工作区的修改时，用命令git checkout -- file。
场景2：当你不但改乱了工作区某个文件的内容，还添加到了暂存区时，想丢弃修改，分两步：
    第一步用命令git reset HEAD file，就回到了场景1，
    第二步按场景1操作。
场景3：已经提交了不合适的修改到版本库时，想要撤销本次提交，参考版本回退一节，不过前提是没有推送到远程库。

远程仓库

  1. git remote add 远程库名(origin) git@github.com:用户名/项目名.git 本地仓库与远程仓关联。
  2. git push -u origin master 把所有内容推送到远程库上。第一次推送加-u，后续只需git push origin master。
  3. git clone git@github.com:用户名/项目名.git。从远程克隆一个本地库。
  4. git checkout -b dev 创建dev分支，并切换到dev分支。-b表示创建并切换。相当于
    • git branch dev
    • git checkout dev
  5. git branch 查看当前分支
  6. git merge dev 将dev分支的工作合并到master分支。
  7. git branch -d dev 删除dev分支。-D强行删除。
  8. git merge –no-ff -m “merge with no-ff” dev 禁用fast forward形式合并分支。
  9. git stash 暂存工作
  10. git stash list 查看暂存区列表。
  11. git stash apply 恢复，stash内容不删除。
  12. git stash drop 删除暂存区内容。
  13. git stash pop 恢复的同时把stash内容也删了。
  14. git remote 查看远程库信息。-v显示更详细的信息。

分支策略

  1. 有新功能或者bug时，新建分支。
  2. master分支是稳定的，仅用来发布新版本，不能在上面干活。
  3. 干活在dev分支上，需要发布新版本时，合并到master分支。
  4. 每个人都在dev分支上干活，每个人都有自己的分支，时不时的往dev分支上合并就行。
  5. 但是，并不是一定要把本地分支往远程推送，那么，哪些分支需要推送，哪些不需要呢？
    * master分支是主分支，因此要时刻与远程同步；
* dev分支是开发分支，团队所有成员都需要在上面工作，所以也需要与远程同步；
* bug分支只用于在本地修复bug，就没必要推到远程了，除非老板要看看你每周到底修复了几个bug；
* feature分支是否推到远程，取决于你是否和你的小伙伴合作在上面开发。
  6. git branch –set-upstream develop origin/develop 指定本地develop分支与远程origin/develop分支的链接。

标签

  1. git tag v1.0 打新标签，默认打在最新提交的commit上。
  2. git tag 查看所有标签
  3. git tag v0.9 commit_id 给指定commit打标签。
  4. git show 查看标签信息。
  5. git tag -d v1.0 删除标签。
  6. git push origin –tags 推送本地标签到远程。
  7. 删除远程标签： 1)git tag -d v0.9 2)git push origin :refs/tags/v0.9
